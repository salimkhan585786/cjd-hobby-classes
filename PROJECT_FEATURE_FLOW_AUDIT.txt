==================================================
2. ROLES
==================================================

Public / guest:
- Can access all public pages
- Can submit contact inquiry
- Can browse courses, gallery, workshops
- Can open art order page
- Cannot access /student or /admin

Student:
- Created through public registration
- Protected route path: /student
- Can access student dashboard, progress, and art order area
- Role source: Firestore users/{uid}.role, fallback is email contains "admin" => admin else student

Admin:
- Not created from public registration
- Must be assigned manually in Firestore users/{uid}.role = "admin"
- Protected route path: /admin
- Can access admin dashboard, manage catalog/workshops/gallery entries/inquiries/announcements/orders overview

4. FEATURE FLOW BY USER TYPE
==================================================

PUBLIC USER FLOW

1. User lands on /
2. Browses courses, gallery, workshops, about, contact
3. If interested in classes:
- goes to /register
- creates student account
- redirected to /student
4. If interested in workshops:
- clicks workshop register CTA
- lands on /contact?topic=workshop
- submits inquiry
- admin later follows up manually
5. If interested in custom art order:
- goes to /order
- fills form
- optionally uploads image
- order saved in Firestore
- Razorpay checkout opens
- on success order marked paid/in-progress

STUDENT FLOW

1. Student registers on /register or logs in at /login
2. After login:
- student redirected to /student
3. Student can:
- view enrolled classes
- view attendance
- view fee status
- view progress
- view workshop registrations
- view announcements
- place custom art order at /student/orders

ADMIN FLOW

1. Admin account must already exist in Firebase Auth
2. Firestore users/{uid}.role must be set to admin
3. Admin logs in at /login
4. Redirect to /admin
5. Admin can:
- seed initial Firestore data
- add/edit/delete courses
- add/delete workshops
- add gallery entries
- review student roster
- review inquiries and mark status
- create announcements
- view fee tracking
- view all orders
- update order lifecycle from /admin/orders

6. RAZORPAY STATUS
==================================================

Razorpay Checkout
Status: PARTIALLY DONE / FRONTEND CONNECTED

What is implemented:
- Loads Razorpay checkout script dynamically
- Creates frontend checkout options
- Opens checkout from art order page
- On success writes paymentId, signature, paidAt, and order status into Firestore
- On dismiss keeps order as pending

What is missing / incomplete:
- No backend order creation using Razorpay secret
- No server-side signature verification
- No webhook handling
- No secure payment confirmation flow
- No retry payment button for existing pending student orders
- No refund/cancel flow

Conclusion:
- Good demo / prototype integration
- Not yet production-safe payment architecture
